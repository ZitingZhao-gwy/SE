# 账户业务子系统测试方案

## 1. 目标

本测试方案面向当前账户业务子系统后端实现，目标是验证以下五类能力：

- 业务规则是否正确
- 账户状态迁移是否正确
- 数据库更新是否一致
- 内外接口链路是否可联调
- 鉴权、幂等、并发等非功能特性是否可接受

本方案参考两份课程材料：

- `docs/SE24(2) (1).ppt`
- `docs/SE25(2) (1).ppt`

其中：

- `SE24` 重点强调面向对象系统的类测试、状态测试、协作测试、线程式集成测试
- `SE25` 重点强调 Web 应用的组件测试、数据库测试、安全测试、性能测试

结合本项目特点，测试重点不放在前端页面，而放在后端业务、接口、数据库和状态流转。

## 2. 系统特点与测试重点

当前系统有四个明显特点：

- 强业务规则：开户、挂失、补办、销户、绑定解绑都有明确前置条件
- 强状态机：`security_account.status` 与 `fund_account.status` 决定大量操作是否合法
- 强协作链：一个操作常常同时修改账户表、日志表、关联关系表
- 强审计性：`fund_transaction_log`、`holding_change_log`、`operation_log` 都必须可追溯

因此测试重点应当是：

- 状态覆盖
- 业务线程覆盖
- 数据一致性覆盖
- 鉴权与幂等覆盖

## 3. 总体测试策略

采用 6 层测试结构：

1. 模型与业务规则测试
2. 状态迁移测试
3. DAO / 数据库一致性测试
4. Service 协作测试
5. Controller / API 集成测试
6. 非功能测试

各层目的如下。

### 3.1 模型与业务规则测试

对应 `SE24` 中的“类测试”和“行为测试”思想。

目标是验证：

- 业务前置条件是否完整
- 错误码是否正确
- 合法输入是否能进入正确业务分支
- 非法输入是否被阻断

### 3.2 状态迁移测试

对应 `SE24` 中的 state coverage。

目标是验证：

- 每个账户状态允许哪些操作
- 非法状态下的操作是否被拒绝
- 合法操作后状态是否迁移到正确结果

### 3.3 DAO / 数据库一致性测试

对应 `SE25` 中的 database testing。

目标是验证：

- 表结构约束是否被正确使用
- 一次业务操作涉及的多张表是否同步更新
- 双向绑定关系是否一致
- 日志事实是否完整落库

### 3.4 Service 协作测试

对应 `SE24` 中的 cluster testing / inter-class testing。

目标是验证：

- service 与多个 DAO 的协作是否正确
- 一个业务线程中的多对象协作是否正确
- 状态、账户、日志、绑定关系是否协同变化

### 3.5 Controller / API 集成测试

对应 `SE25` 中的 component-level testing。

目标是验证：

- HTTP 接口请求与响应是否符合文档
- 鉴权头和 token 是否生效
- DTO 校验与业务错误码是否正确
- 外部接口和内部接口是否能完成联调闭环

### 3.6 非功能测试

目标是验证：

- 鉴权安全
- 幂等控制
- 并发一致性
- 性能与负载承受能力

## 5. 需要重点覆盖的状态

### 5.1 证券账户状态

- `NORMAL`
- `LOSS_FROZEN`
- `VIOLATION_FROZEN`
- `NO_FUND_FROZEN`
- `PRE_CLOSE`
- `CLOSED`

重点验证以下迁移：

- `NORMAL -> LOSS_FROZEN`
- `LOSS_FROZEN -> CLOSED + 新证券账户 NORMAL`
- `NORMAL -> CLOSED`
- `NORMAL -> NO_FUND_FROZEN`
- `NO_FUND_FROZEN -> NORMAL`
- `VIOLATION_FROZEN -> NORMAL`

### 5.2 资金账户状态

- `NORMAL`
- `LOSS_FROZEN`
- `VIOLATION_FROZEN`
- `CLOSED`

重点验证以下迁移：

- `NORMAL -> LOSS_FROZEN`
- `LOSS_FROZEN -> CLOSED + 新资金账户 NORMAL`
- `NORMAL -> CLOSED`
- `VIOLATION_FROZEN -> NORMAL`

## 6. 测试分类与内容

## 6.1 业务规则测试

这是第一优先级。

建议至少覆盖以下场景：

- 未成年人证券开户失败
- 非法证件类型或非法证件号码开户失败
- 黑名单投资者开户失败
- 同一投资者重复证券开户失败
- 证券账户有持仓时不能销户
- 资金账户有可用余额或冻结余额时不能销户
- 资金账户挂失会联动冻结证券账户
- 资金账户销户后证券账户进入 `NO_FUND_FROZEN`
- 重新绑定资金账户后证券账户恢复 `NORMAL`
- 投资者信息修改后数据正确落库
- 工作人员停用后不能继续使用原 token

## 6.2 状态迁移测试

建议做成一张状态迁移矩阵。

每条用例至少记录：

- 当前状态
- 执行操作
- 预期结果
- 目标状态
- 预期错误码

示例：

| 对象 | 当前状态 | 操作 | 预期 |
|---|---|---|---|
| `security_account` | `NORMAL` | 挂失 | 进入 `LOSS_FROZEN` |
| `security_account` | `LOSS_FROZEN` | 补办 | 旧账户 `CLOSED`，新账户 `NORMAL` |
| `fund_account` | `NORMAL` | 销户 | 若余额为 0，则 `CLOSED` |
| `fund_account` | `VIOLATION_FROZEN` | 管理解冻 | 恢复 `NORMAL` |

## 6.3 数据库一致性测试

重点检查以下一致性：

- 证券开户后，`investor` 与 `security_account` 是否同时正确写入
- 资金开户后，`fund_account.sec_acc_no` 与 `security_account.linked_fund_acc` 是否一致
- 解绑后，双向关系是否同时清空
- 资金销户后，证券账户是否转成 `NO_FUND_FROZEN`
- 交易回调后，余额与持仓是否同步更新
- `ref_order_id` 是否能把 `fund_transaction_log` 与 `holding_change_log` 关联起来
- 柜台操作、管理员操作是否都写入 `operation_log`

## 6.4 Service 协作测试

建议按“业务线程”而不是按孤立方法测试。

最关键的线程有：

### 线程1：证券开户 -> 资金开户

检查：

- 是否创建投资者
- 是否创建证券账户
- 是否创建资金账户
- 是否自动绑定
- 是否记录操作日志

### 线程2：资金挂失 -> 资金补办

检查：

- 旧资金账户状态是否正确
- 证券账户是否联动冻结
- 新资金账户是否创建
- 旧 token 是否失效
- 绑定关系是否转移到新账户

### 线程3：买入链路

流程：

- `updateFundBalance(买入冻结)`
- `updateFundBalance(买入扣款)`
- `updateSecurityHolding(买入增加)`

检查：

- 可用余额和冻结余额变化
- 当前持仓数量与均价变化
- 两张日志表都写入
- `ref_order_id` 可关联

### 线程4：卖出链路

流程：

- `updateSecurityHolding(卖出冻结)`
- `updateSecurityHolding(卖出扣减)`
- `updateFundBalance(卖出回款)`

检查：

- 冻结持仓变化
- 总持仓变化
- 资金回款变化
- 日志联动

### 线程5：资金销户 -> 重新绑定

检查：

- 资金账户是否 `CLOSED`
- 证券账户是否进入 `NO_FUND_FROZEN`
- 重新绑定后是否恢复 `NORMAL`

## 6.5 API 集成测试

接口测试建议按以下类型组织：

### 内部接口

- 工作人员登录
- 证券开户 / 信息修改 / 挂失 / 补办 / 销户
- 资金开户 / 存款 / 取款 / 改密 / 挂失 / 补办 / 销户 / 绑定解绑
- 管理员冻结 / 解冻 / 强制销户 / 年度结息
- 审计日志查询

### 外部接口

- 投资者登录
- 资金快照查询
- 持仓快照查询
- 投资者改密
- 交易系统资金回调
- 交易系统持仓回调

### 集成测试应验证的要点

- HTTP 方法与 URL 是否正确
- 请求参数名是否与 README 一致
- DTO 参数校验是否生效
- `X-Staff-Auth-Token` 是否生效
- `auth_token` 是否生效
- 返回字段是否符合预期
- 错误码是否正确

## 6.6 安全测试

建议最少覆盖以下安全点：

- 无工作人员 token 调用 `/api/admin/**`
- 错误工作人员 token 调用 `/api/admin/**`
- 无投资者 token 调用外部查询接口
- 投资者 token 查错账户是否被拒绝
- 已停用工作人员的旧 token 是否失效
- 密码错误时是否返回正确错误码
- 交易回调重复调用是否被幂等机制挡住

说明：

当前系统没有 RBAC，因此管理员接口本质上是“有效工作人员 token 即可调用”。这不是 bug，而是当前设计决策。测试时要明确区分：

- 已实现：工作人员身份鉴权
- 未实现：管理员角色权限控制

## 6.7 性能与并发测试

当前没有前端，因此性能测试重点放在接口和数据库。

建议至少做以下测试：

- `getFundSnapshot` 并发查询
- `getSecuritySnapshot` 并发查询
- `updateFundBalance` 重复并发调用
- `updateSecurityHolding` 重复并发调用
- 审计日志查询大时间范围测试

重点关注：

- 平均响应时间
- 高并发下数据是否错乱
- 幂等是否失效
- 日志是否重复写入
- 查询性能是否明显下降

## 7. 工具建议

建议使用以下工具组合：

- `JUnit 5`：业务规则测试、service 测试
- `H2` 或测试数据库：DAO / 数据一致性测试
- `MockMvc`：Controller / API 集成测试
- `curl` / Postman / Apifox`：联调脚本与人工验收
- `JMeter`：压力测试
- `JaCoCo`：覆盖率报告

## 8. 优先级建议

建议按照以下顺序推进：

### 第一阶段

- 业务规则测试
- 状态迁移测试

原因：

- 这是系统正确性的核心
- 也是课程检查时最容易被问到的部分

### 第二阶段

- API 集成测试
- 数据库一致性测试

原因：

- 这决定系统能否联调
- 也能证明 README 与代码一致

### 第三阶段

- 安全测试
- 并发 / 性能测试

原因：

- 这是质量加分项
- 适合在主流程稳定后进行

## 10. 对本项目的最终测试建议

这个项目最值得强调的不是 UI，而是：

- 状态覆盖
- 业务线程覆盖
- 数据一致性
- 鉴权正确性
- 日志可追溯性

因此后续测试工作的主轴建议定为：

1. 先补业务规则与状态迁移测试
2. 再补接口线程集成测试
3. 再补数据库一致性检查
4. 最后做安全与性能测试

这样最符合当前项目形态，也最符合 `SE24` 和 `SE25` 两份课件的核心要求。
