## <a name=''></a>1. 引言

### <a name='-1'></a>1.1. 编写目的

本测试报告用于验证证券账户与资金账户管理子系统的功能正确性、业务规则正确性、接口可用性、安全性及稳定性。通过对系统各模块进行功能测试、边界测试、压力测试、接口测试和安全性测试，评估系统是否满足设计要求，并为后续联调与验收提供依据。

本项目涉及的相关人员与子系统有：账户管理人员，交易系统管理子系统，交易客户端子系统

### <a name='-1'></a>1.2. 项目背景

本项目为证券交易系统中的账户业务子系统，负责证券账户与资金账户的生命周期管理，并为其他交易系统提供账户状态、资金信息、账户信息更新及持仓信息支持。

### <a name='-1'></a>1.3. 术语与缩写解释

| 术语               | 含义     |
| ---------------- | ------ |
| Security Account | 证券账户   |
| Fund Account     | 资金账户   |
| Investor         | 投资者    |
| Staff            | 工作人员   |
| Auth Token       | 认证令牌   |
| Holding          | 持仓     |
| Operation Log    | 操作日志   |
| DAO              | 数据访问层  |
| DTO              | 数据传输对象 |

### <a name='-1'></a>1.4. 系统概述

对最终集成的整体系统与本组的负责子系统分别进行概述，说明具体功能要求

本项目源于浙江大学 **2026夏学期 软件工程基础** 课程，目的在于构建一个完整的股票交易系统，主要包括六大业务：证券账户业务、资金账户业务、交易客户端、股票中央交易系统、网上信息发布、交易系统管理。

其中我们组负责：证券账户业务、资金账户业务。功能主要有：
| 业务模块   | 功能描述                            |
| ------ | ------------------------------- |
| 证券账户管理 | 支持证券账户的开户、挂失、补办及销户等生命周期管理操作。    |
| 资金账户管理 | 支持资金账户的开户、存取款、密码修改、挂失、补办及销户等操作。 |
| 查询服务   | 支持资金账户余额及证券账户持仓信息查询。            |
| 身份认证   | 支持投资者和工作人员的身份认证与登录。             |
| 交易服务  | 支持交易客户端调用接口调用来返回交易信息  。             |

> 具体功能将会在本报告后半部分呈现

### <a name='-1'></a>2.2. 工作人员管理功能测试

#### <a name='-1'></a>2.2.1. 测试目标与范围

验证工作人员登录认证、访问鉴权、工作人员停用、令牌失效和操作日志查询功能。

#### <a name='-1'></a>2.2.2. 测试用例

| 用例编号 | 测试功能 | 重要级别 | 预置条件 | 输入与操作步骤 | 预期结果 |
| --- | --- | --- | --- | --- | --- |
| STAFF-001 | 工作人员正常登录 | 高 | 数据库存在状态为“正常”的工作人员 | 输入正确用户名和密码登录 | 返回成功；响应包含工作人员编号、用户名、正常状态和非空认证令牌 |
| STAFF-002 | 登录密码错误 | 高 | 工作人员存在且状态正常 | 使用正确用户名、错误密码登录 | 返回 `ERR_009`；不签发令牌 |
| STAFF-003 | 不存在的用户登录 | 中 | 用户名不存在 | 使用不存在的用户名登录 | 返回 `ERR_009`，且不泄露用户名是否存在等额外信息 |
| STAFF-004 | 禁用人员登录 | 高 | 目标工作人员状态为“禁用” | 使用其正确用户名和密码登录 | 返回 `ERR_009`；禁止登录 |
| STAFF-005 | 登录参数校验 | 中 | 无 | 分别提交空用户名、空密码和缺少字段的请求 | 返回 `ERR_PARAM`；不进入登录业务 |
| STAFF-006 | 工作人员令牌鉴权 | 高 | 正常登录并取得工作人员令牌 | 携带令牌执行证券开户等受保护操作 | 鉴权通过，系统进入业务处理 |
| STAFF-007 | 缺失或伪造令牌 | 高 | 无有效令牌 | 不带令牌或携带随机令牌执行受保护操作 | 返回 `ERR_018`；数据库无业务数据变更 |
| STAFF-008 | 停用工作人员 | 高 | 操作人员已登录；目标人员状态正常 | 提交目标工作人员停用申请 | 目标状态变为“禁用”；生成“停用工作人员”操作日志 |
| STAFF-009 | 停用后旧令牌失效 | 高 | 目标人员已登录并持有令牌 | 停用目标人员后，继续使用旧令牌执行受保护操作 | 返回 `ERR_018`；旧令牌不可继续访问 |
| STAFF-010 | 重复停用 | 中 | 目标人员已被停用 | 再次提交停用申请 | 返回 `ERR_011`；不重复写入有效状态变更 |
| STAFF-011 | 操作日志查询 | 高 | 已完成开户、存取款、挂失等操作 | 按人员、对象或时间条件查询操作日志 | 返回与筛选条件匹配的日志；日志包含操作人员、操作类型、对象、详情和时间 |
| STAFF-012 | 工作台统计及最近日志 | 中 | 系统已有账户和操作日志 | 查看账户统计和最近操作日志 | 账户统计与数据库一致；最近日志按预期顺序和条数返回 |

## <a name='-1'></a>4. 压力测试

### <a name='-1'></a>4.1. 测试目标

压力测试用于评估账户业务子系统在并发访问和连续交易更新条件下的响应能力、稳定性及数据一致性。测试重点包括：

- 资金账户快照查询的并发处理能力；
- 证券账户及持仓快照查询的并发处理能力；
- 连续资金变更和持仓变更的处理能力；
- 同一业务请求并发重复提交时的幂等性；
- 负载升高后系统响应时间、吞吐量和错误率的变化；
- 高负载结束后账户余额、持仓和日志是否保持一致。

本章只制定测试方案和结果记录格式。实际结果必须在真实执行后填写，不使用单元测试耗时或模拟数据代替压力测试结果。

### <a name='-1'></a>4.2. 测试环境与控制

#### <a name='-1'></a>4.2.1. 测试工具

建议使用 Apache JMeter 或同类负载测试工具生成并发请求，使用 MySQL 查询工具核对业务数据，并使用操作系统监控工具记录服务器 CPU、内存和网络使用情况。

#### <a name='-1'></a>4.2.2. 环境控制

为保证各轮结果可比较，测试期间应保持以下条件一致：

- 使用相同版本的后端程序、JDK 和 MySQL；
- 后端服务与数据库的部署位置和硬件资源保持不变；
- 每轮测试使用同规模的基础账户、持仓和日志数据；
- 正式采样前先进行一次预热，避免首次加载影响结果；
- 每轮开始前恢复测试数据，避免余额、持仓和日志累积干扰后续轮次；
- 压测工具与被测服务尽量部署在不同设备，避免资源竞争；
- 测试期间停止与本测试无关的批处理任务和大规模数据库操作；
- 记录测试时间、程序版本、数据库版本及设备配置。

### <a name='-1'></a>4.3. 负载模型与输入

采用逐级加压方式，每个阶段持续5分钟，阶段之间预留1分钟观察和恢复时间。正式测试前使用10个并发用户预热2分钟。

| 阶段 | 并发用户数 | 启动时间 | 持续时间 | 目的 |
| --- | ---: | ---: | ---: | --- |
| 预热 | 10 | 30秒 | 2分钟 | 完成连接、类和缓存预热，不计入正式结果 |
| 基准负载 | 50 | 30秒 | 5分钟 | 验证正常并发下的响应能力 |
| 中等负载 | 100 | 60秒 | 5分钟 | 观察吞吐量和响应时间变化 |
| 高负载 | 200 | 120秒 | 5分钟 | 验证高并发稳定性并识别性能拐点 |

测试输入数据要求如下：

- 准备不少于200组正常且已绑定的资金账户、证券账户和认证令牌；
- 每个证券账户至少准备5只证券持仓，部分持仓包含冻结数量；
- 用于资金变更的账户应具有足够的可用余额或冻结余额；
- 用于持仓变更的账户应具有足够的可卖数量或冻结数量；
- 正常交易变更使用唯一业务流水号；
- 幂等测试中的并发请求使用相同业务流水号和相同变更类型；
- 各场景使用独立测试账户，避免不同线程修改同一账户造成非预期干扰。

### <a name='-1'></a>4.4. 压力测试场景

| 场景编号 | 测试场景 | 负载方式 | 主要检查点 |
| --- | --- | --- | --- |
| PERF-001 | 资金账户快照并发查询 | 按50、100、200并发逐级执行 | 查询成功率、响应时间、吞吐量；余额和流水内容正确 |
| PERF-002 | 证券账户快照并发查询 | 按50、100、200并发逐级执行；随机查询全部持仓或单只证券 | 查询成功率、响应时间、吞吐量；持仓数量和冻结数量正确 |
| PERF-003 | 连续资金变更 | 使用不同账户和唯一业务流水号持续提交资金冻结、扣款、解冻或回款 | 余额无负数；每次变更只产生一条对应流水；无数据丢失 |
| PERF-004 | 连续持仓变更 | 使用不同账户持续提交持仓增加、冻结、扣减或释放 | 总持仓、冻结持仓和可卖数量满足业务约束；日志完整 |
| PERF-005 | 资金变更并发幂等 | 多个线程同时提交完全相同的资金变更 | 仅一次请求实际修改余额，其余请求识别为重复；仅产生一条有效流水 |
| PERF-006 | 持仓变更并发幂等 | 多个线程同时提交完全相同的持仓变更 | 仅一次请求实际修改持仓，其余请求识别为重复；仅产生一条有效日志 |
| PERF-007 | 混合业务负载 | 资金快照40%、证券快照40%、资金变更10%、持仓变更10% | 混合负载下系统无崩溃、死锁或大面积超时，数据保持一致 |
| PERF-008 | 峰值后恢复 | 高负载结束后降至10并发继续运行2分钟 | 响应时间能够恢复，无连接泄漏和持续性错误 |

每轮测试结束后，应核对资金余额、冻结资金、持仓数量、冻结持仓以及对应日志。不能只依据请求成功率判断测试通过。

### <a name='-1'></a>4.5. 评价指标与建议验收标准

| 指标 | 含义 | 建议验收标准 |
| --- | --- | --- |
| 请求成功率 | 业务成功或符合预期的幂等响应占全部请求的比例 | 50和100并发时不低于99%；200并发时不低于98% |
| 平均响应时间 | 全部请求响应时间的平均值 | 查询类操作不超过500毫秒；更新类操作不超过800毫秒 |
| P95响应时间 | 95%的请求可在该时间内完成 | 查询类操作不超过1秒；更新类操作不超过1.5秒 |
| P99响应时间 | 99%的请求可在该时间内完成 | 不超过3秒 |
| 吞吐量 | 单位时间内完成的请求数 | 随并发增加不应在低负载阶段明显下降，具体基线以首次实测为准 |
| 业务错误率 | 非预期业务错误和系统错误占比 | 系统内部错误为0；非预期业务错误不超过1% |
| 幂等正确率 | 重复请求未造成重复数据变更的比例 | 100% |
| 数据一致性 | 余额、持仓、绑定关系和日志相互一致 | 100%，不得出现负余额、负持仓、重复流水或日志缺失 |
| 资源使用率 | CPU、内存和数据库连接使用情况 | 无持续性100%占用、内存持续增长或连接耗尽 |

以上数值为课程项目的建议验收标准。正式执行前可根据测试设备性能调整，但调整值和原因必须记录，不能在测试完成后为适配结果而修改标准。

## <a name='-1'></a>5. 其他模块接口测试

### <a name='-1'></a>5.1. 测试目标

本测试用于验证交易客户端与账户业务子系统之间的数据交互是否正确，重点检查客户端登录、资金快照、证券快照、资金变更和持仓变更五类联调场景。

联调测试不仅检查客户端收到的业务结果，还应核对账户状态、余额、持仓和日志是否同步变化，保证跨模块传递的数据完整、准确且可追溯。

### <a name='-1'></a>5.2. 测试对象与范围

本章测试对象为交易客户端与账户业务子系统之间的交互，范围如下：

- 客户端使用资金账户和交易密码完成登录；
- 客户端查询资金余额、冻结资金、账户状态和最近流水；
- 客户端查询证券账户状态及全部或单只证券持仓；
- 外部业务更新资金账户的可用余额和冻结余额；
- 外部业务更新证券账户的总持仓和冻结持仓。

### <a name='-1'></a>5.3. 测试控制与准备

联调前应完成以下准备：

- 部署相同版本的交易客户端、账户业务服务和数据库脚本；
- 准备正常、冻结、未绑定和已销户等不同状态的资金及证券账户；
- 为正常账户设置已知交易密码，并准备正确与错误两组密码；
- 为快照查询准备包含多只证券、冻结持仓及资金流水的账户；
- 为资金变更准备足够的可用余额和冻结余额；
- 为持仓变更准备足够的可卖数量和冻结持仓；
- 每个正常变更场景使用唯一业务流水号，重复性测试使用相同业务流水号；
- 每个场景执行前记录初始数据，执行后立即核对返回结果和数据库；
- 各用例使用独立账户或在执行后恢复数据，避免相互影响。

建议按“登录—快照查询—资金变更—持仓变更—再次查询”的顺序执行，以验证一次完整的数据交互闭环。

### <a name='-1'></a>5.5. 数据一致性检查

- 登录返回的资金账户与证券账户绑定关系和数据库一致；
- 资金快照与资金账户余额、冻结余额和最近流水一致；
- 证券快照满足“可卖数量=总持仓数量-冻结数量”；
- 资金变更后的余额与资金流水中的变更后余额一致；
- 持仓变更后的数量与持仓变更日志一致；
- 同一业务流水号的重复请求不会生成重复有效记录；
- 失败场景不会产生部分更新或残留日志；
- 变更完成后重新查询快照，能够观察到最新且一致的数据。

## <a name='-1'></a>6. 安全性测试

### <a name='Session'></a>6.1. Session认证测试

Session认证测试用于验证登录状态能否被安全地建立、保持和终止。当前后端代码主要使用认证令牌，尚未发现完整的 HTTP Session 实现，因此本节作为系统集成后的待验证项，不将其表述为已经实现的能力。

| AUTH-003 | 停用后旧令牌失效 | 工作人员已登录并持有令牌 | 停用该工作人员后继续使用原令牌 | 原令牌失效；不能继续执行受保护操作 |
| AUTH-004 | 缺失或伪造客户端令牌 | 资金及证券账户存在 | 不携带令牌或使用随机令牌查询账户数据 | 返回 `ERR_018`；不泄露余额和持仓 |
| AUTH-005 | 跨资金账户越权 | A、B账户均存在且A已登录 | 使用A的令牌查询B的资金账户 | 返回 `ERR_018`；不返回B账户数据 |
| AUTH-006 | 跨证券账户越权 | A、B账户均存在且A已登录 | 使用A的令牌查询B的证券账户 | 返回 `ERR_018`；不返回B账户持仓 |
| AUTH-007 | 补办或销户后令牌失效 | 账户已登录并持有令牌 | 完成账户补办或销户后继续使用旧令牌 | 返回 `ERR_018`；旧账户会话不能继续使用 |
| AUTH-008 | 直接访问受保护功能 | 用户未完成认证 | 绕过正常页面流程直接访问受保护功能 | 后端独立完成鉴权并拒绝访问，不能只依赖前端页面限制 |

| 用例编号 | 测试场景 | 预置条件 | 输入与操作步骤 | 预期结果 |
